왜 선택했나?
GAS (Gameplay Ability System)
스킬 추가 시 캐릭터 클래스를 직접 수정하지 않아도 되는 구조가 필요했습니다.
GameplayTag 기반 상태 제어로 코드 결합도를 낮추고,
쿨타임·비용·이펙트를 하나의 Ability 단위로 캡슐화했습니다.
왜 선택했나?
Weapon System
다양한 무기 확장과 유지보수를 고려해 무기 시스템을 모듈화된 구조로 설계했습니다.
무기 교체, 공격 방식 변경을 데이터 기반으로 처리하여 코드 수정 없이 새로운 무기를 추가할 수 있도록 하는 것을 목표로 했습니다.
왜 선택했나?
Tag 기반 상태 관리
bool 변수로 상태를 관리하면 조건 조합이 폭발적으로 증가합니다.
GameplayTag 계층 구조로 상태 추가·삭제가 명확하고, 태그 쿼리만으로 조건 분기가 가능합니다.
왜 선택했나?
Enhanced Input System
레거시 Input Binding은 컨텍스트 전환(UI, Combat, Cutscene)이 불편합니다.
InputMappingContext를 활용해 상태별 입력 레이어를 런타임에 추가·제거하는 구조로 설계했습니다.
03 — Deep Dive

핵심 구현

3가지 핵심 시스템의 구현 방식을 코드,이미지지와 함께 설명합니다.

WeaponSystem 구조

런타임에 현재 장착중인 무기와 상태, 입력에 따라 콤보 공격을 선택합니다.

  • UK_WeaponData : 캐릭터의 무기를 카테고리 별로 등록
  • UK_StatusAnimData : 무기 부착 관련 정보와 상태 기반 TMap검색을 통한 UK_AnimData를 등록
  • UK_AnimData : 캐릭터가 출력할 몽타주를 등록
  • 플레이어의 입력 & 캐릭터의 상태에 따른 애니메이션

Ability System 설계

ASC를 PlayerState에 배치하고, 리스폰 시에도 어빌리티·어트리뷰트가 유지되도록 설계했습니다.

  • ASC는 PlayerState, Avatar는 Character로 분리
  • SetByCaller로 런타임 데미지 값을 직접 전달
  • GameplayTag로 Cooldown/Cost Effect 명확히 구분
  • GameplayCue 분리로 클라이언트 이펙트 독립 처리
// SetByCaller를 통한 데미지 전달
void UGA_MeleeAttack::ApplyDamage(
    AActor* Target)
{
    FGameplayEffectSpecHandle SpecHandle =
        MakeOutgoingGameplayEffectSpec(
            DamageEffectClass,
            GetAbilityLevel());

    SpecHandle.Data->SetSetByCallerMagnitude(
        TAG_Data_Damage,
        BaseDamage * ComboMultiplier);

    ApplyGameplayEffectSpecToTarget(
        SpecHandle, GetASC(Target));
}
// Tag 기반 어빌리티 실행 조건
bool UGA_MeleeAttack::CanActivateAbility(
    ...) const
{
    if (HasMatchingGameplayTag(TAG_State_Stunned)
     || HasMatchingGameplayTag(TAG_State_Dead))
    {
        return false;
    }
    return Super::CanActivateAbility(...);
}

커스텀 Movement 처리

UCharacterMovementComponent를 상속해 상태별 이동 파라미터를 동적으로 변경하는 구조로 설계했습니다.

  • Sprint / Walk / Combat Stance별 MaxWalkSpeed 분리
  • 회전 처리: UseControllerDesiredRotation + Yaw 보간
  • 회피 도중 입력 버퍼링으로 연속 동작 지원
  • 거리 기반 Tick 주기 조절로 성능 절약
// 상태 기반 이동 속도 설정
void UMyCharMovement::UpdateMovementParams()
{
    const bool bSprinting =
        HasTag(TAG_State_Sprinting);
    const bool bInCombat =
        HasTag(TAG_State_Combat);

    if      (bSprinting) MaxWalkSpeed = 600.f;
    else if (bInCombat)  MaxWalkSpeed = 300.f;
    else                 MaxWalkSpeed = 450.f;

    bOrientRotationToMovement     = !bInCombat;
    bUseControllerDesiredRotation =  bInCombat;
}

05 — Problem Solving

문제 해결 경험

실제 겪은 버그와 분석·해결 과정입니다. 이게 진짜 실무 능력입니다.

01 스킬 쿨타임이 적용되지 않는 문제 GAS ▾
현상
스킬 사용 후 쿨타임 없이 즉시 재사용 가능. HUD의 쿨타임 UI는 정상 작동.
원인 분석
ASC를 Character에 배치했을 때, 리스폰 시 캐릭터가 Destroy → Spawn되면서 ASC도 함께 초기화됨. 쿨타임 GameplayEffect가 유지되지 않았던 것. GetActivatableAbilities()로 런타임 상태 확인 후 원인 특정.
해결
ASC를 PlayerState로 이동. Character는 Avatar Actor만 담당하도록 분리. InitAbilityActorInfo(PlayerState, Character)로 재연결. 리스폰 후에도 ASC·어트리뷰트 유지 확인.
02 Tick 비용이 전투 진입 시 2배 이상 급증 Performance ▾
현상
일반 상태에서는 정상이나, 전투 진입 시 GameThread Tick이 2배 이상 상승.
원인 분석
dumpticks 실행 결과, AttackTraceComponent가 매 Tick SphereTrace를 실행하고 있음을 발견. 전투 진입 시 무조건 Trace가 활성화되는 로직 오류.
해결
Ability의 ActivateAbility에서 Trace 시작, EndAbility에서 Trace 중단. Tick 자체를 Disable하고 필요 시 수동 호출 방식으로 전환. GameThread Tick 18.4ms → 11.4ms (38% 절감).
03 PlayerStart 미설정 시 캐릭터가 맵 밖에 스폰 Gameplay ▾
현상
특정 맵 로드 후 캐릭터가 월드 원점(0,0,0) 또는 맵 밖에 스폰.
원인 분석
GameMode의 FindPlayerStart가 특정 조건에서 null 반환. PlayerStart 태그 필터링 로직에서 태그 미설정 시 폴백 처리 누락.
해결
ChoosePlayerStart_Implementation 오버라이드. 유효한 PlayerStart 없을 경우 Safety Spawn으로 폴백. 태그 검색 실패 시 로그 출력으로 조기 탐지.
